# llamastash `init` Wizard — v2 Requirements

> Companion to [`docs/brainstorms/llamatui-requirements.md`](./llamatui-requirements.md) (v1 R1–R47). This document covers a single v2 feature: the first-run setup wizard. IDs continue from the v1 doc to keep references globally unambiguous across the project.

## Problem Frame

A fresh user installing llamastash today hits a wall before they ever see the TUI: `llama-server` from llama.cpp is a hard runtime dependency (R10), and locating, installing, or building the right variant for their GPU is non-trivial. Even users who get past that step then have to know *which* GGUFs to download for their hardware, where to put them, and what `llama-server` flags will get the most out of their machine. v1's success criterion — "launch the TUI and have a healthy endpoint within ~10 seconds *without reading docs*" — silently assumes a working `llama-server` binary and at least one GGUF on disk. For users without either, the path from `cargo install llamastash` to first chat is a documentation maze.

The `init` wizard closes that gap. It is the front door for users who arrive with nothing: detects hardware, installs the right llama.cpp build for their GPU, recommends a starter GGUF tuned to their VRAM, downloads it into the HuggingFace cache layout, writes a config tuned to their architecture, and verifies the whole chain works before declaring success. It must also be re-runnable as a maintenance tool (GPU swaps, llama.cpp upgrades, model re-recommendations) — not a one-shot bootstrap.

**Audience expansion is a deliberate strategic move, not a continuation.** v1's primary audience is "developers, ML practitioners, and power-user enthusiasts comfortable at the terminal" — people who already have llama.cpp or can `brew install llama.cpp` in one line. `init`'s primary audience is the *adjacent* user: a developer who wants a local LLM but isn't llama.cpp-fluent. This opens llamastash to a broader funnel that overlaps with Ollama's user base. The product accepts that overlap and bets its differentiation for that audience on transparency, agent-friendliness, and GGUF-native intelligence — not on ease-of-onboarding alone. (See Key Decisions for the curated-recommender identity commitment that follows from this expansion.)

Audience:
- Primary: a first-time llamastash user on a fresh dev machine with no llama.cpp and no GGUFs.
- Secondary: existing users running maintenance — "did llama.cpp drift?", "regenerate my config for a new GPU", "show me a better model for my hardware now".
- Tertiary: agents and CI driving `init --yes --json` for unattended setup.

## Requirements

**Wizard Flow & Surface**

- **R48.** `init` is a top-level subcommand: `llamastash init`. It is a terminal-prompts-only flow (stdin/stdout); no ratatui screens. Prompts use an interactive prompt library (e.g. `dialoguer`-style arrow-key select + confirm + input).
- **R49.** Default step order, all skippable: (1) hardware + existing-install detection, (2) install or locate `llama-server`, (3) recommend and download a starter GGUF, (4) write a tuned `config.yaml` (with an optional in-step theme selection sub-prompt), (5) post-install smoke launch, (6) handoff. Each step prints a one-line state summary on completion.
- **R50.** A persistent header (or initial summary block) shows the detected hardware context for the duration of the wizard. Each prompt that depends on a specific hardware property surfaces just that property next to the recommendation (e.g. the VRAM number next to the model recommender, the GPU vendor next to the install picker), so the user can audit and override without visual clutter.

**llama.cpp / `llama-server` Installation**

- **R51.** Install-method prompt offers, in order: (1) hardware-aware default with a short `[reason]` in brackets, (2) Homebrew — shown **only** if brew is detected and is not the same path as #1, (3) "I'll point at an existing `llama-server` binary" — writes the absolute path to `llama_server_path` in `~/.config/llamastash/config.yaml`.
- **R52.** Hardware-aware default policy, branched explicitly per platform × detected GPU class (R44):
  - **macOS Apple Silicon (M-series)** → Homebrew Metal bottle. Metal works; brew upgrades are trivial.
  - **macOS Intel** → Homebrew CPU bottle. Per v1 R44, Intel macOS is CPU-only-assumed; installing the Metal bottle is wasted bytes and may not load.
  - **Linux + NVIDIA detected** → `ggml-org/llama.cpp` GitHub Releases CUDA prebuilt.
  - **Linux + AMD/ROCm detected** → GitHub Releases HIP prebuilt.
  - **Linux + Vulkan-only detected** (R44 returns `GpuInfo::Unknown` because Vulkan can't reliably report vendor/memory) → GitHub Releases Vulkan prebuilt, with an inline detection note in the prompt so the user can override to CPU-only if Vulkan isn't desired.
  - **Linux CPU-only** → GitHub Releases CPU prebuilt. (Brew is not preferred on Linux even when installed: Linuxbrew is uncommon and the Releases path is a single source of truth that the wizard's verification contract already trusts.)
  - **Invariant:** never silently install a CPU-only build on a machine where R44 reports a usable GPU.
- **R53.** When the GitHub Releases path is taken, llamastash downloads the appropriate platform+variant archive over HTTPS and verifies its integrity before extracting. Verification contract: (a) checksums are fetched via the GitHub Releases REST API (not scraped from rendered HTML); (b) algorithm is SHA-256 or stronger; (c) verification happens before extraction — on mismatch or unavailable checksum, the wizard aborts the install with an actionable message and offers the "point at an existing binary" path instead of proceeding unverified; (d) the verified digest is recorded in `~/.config/llamastash/config.yaml` alongside `llama_server_path` so `doctor` (R74) can detect post-install tampering. Extraction target is a versioned subdirectory under `$XDG_DATA_HOME/llamastash/llama-cpp/<version>/` (mode `0700`); the archive is rejected if it contains entries with `..`, absolute paths, device/FIFO entries, or symlinks pointing outside the extract root; extract uses `<dir>.tmp.<pid>` + atomic rename. The resulting `llama-server` binary is `chmod 0700`.
- **R54.** Existing-install detection runs before the install-method prompt: probe `$PATH`, `LLAMASTASH_LLAMA_SERVER`, common locations (`/opt/homebrew/bin`, `/usr/local/bin`, prior llamastash-managed install dir). If a working binary is found, surface it as a pre-selected "use existing" option with version + path shown; user can still pick a fresh install.

**Model Recommendation**

- **R55.** A Rust-native model recommender module (single internal module boundary, not yet a published crate; referred to throughout this document as "the recommender") reuses R7 (GGUF metadata), R8 (KV-cache-aware VRAM estimate), and R44 (GPU detection). Algorithmic depth follows whichllm's approach but ports only the consumer side: VRAM-fit hard filter (with explicit safety margin: estimated peak ≤ 0.90 × detected VRAM minus a backend-specific overhead band — configurable, with sensible per-backend defaults), then composite ranker over a benchmark score, tok/s estimate (bandwidth × quant efficiency × backend factor), parameter-count quality curve, and recency. The candidate universe is filtered to **chat-capable GGUFs** (R70 phase-2 sends `/v1/chat/completions`); embed/rerank models are out of scope for init's first-launch experience. The recommender's candidate universe IS the benchmark-snapshot index (R56) — it does not query the HuggingFace search API separately. The recommender is one component; the bundled+remote benchmark snapshot (R56) is its data input, not a separate engine. **VRAM-unknown branch (R44 returns `GpuInfo::Unknown`, e.g. Vulkan-only Linux):** the recommender cannot apply the VRAM-fit filter. The wizard prompts the user for an approximate VRAM number (with the Vulkan device name shown for context) before invoking the recommender, falling back to the CPU-class size bucket (1–3B Q4) if the user declines to estimate.
- **R56.** Benchmark data is sourced as a single composite JSON snapshot. Two-tier delivery: (a) a snapshot bundled into the binary at compile time, refreshed per llamastash release, used as fallback; (b) a fresher copy hosted at a stable URL (preference: a GitHub Releases asset to keep main-branch git history clean), regenerated daily by a GitHub Actions job. At init time, the recommender attempts a short-timeout fetch of the remote copy and falls back to the bundled snapshot on failure or air-gapped environments. Snapshot staleness is measured **relative to the bundle-date stamp inside the snapshot itself**, not against today's wall-clock; this anchors `doctor`'s drift threshold (R74) to a reproducible reference. Fetch contract: (1) HTTPS only, no plaintext fallback; (2) host allowlist restricted to `raw.githubusercontent.com` / `objects.githubusercontent.com` / `github.com` release CDN; (3) maximum 3 redirects, cross-host redirects refused, redirects whose resolved IP is private / loopback / link-local / metadata-service (169.254.169.254) refused; (4) total response body size capped (e.g. 5 MB); (5) the remote snapshot's integrity is verified — either via an embedded signature or via a monotonic-timestamp + minimum-version check committed in the bundled snapshot — before it is allowed to override the bundle. Verification failure is silent (use bundled), not loud (use remote).
- **R57.** The benchmark-snapshot generation script lives in `scripts/` (Python is acceptable since it runs only in CI, not in the binary). It may use whichllm's MIT-licensed `benchmark_sources/` code with attribution recorded in `NOTICE`, which also records the upstream commit hash for the currently vendored revision so the sync trail is auditable. Upstream sync is **on-demand** (triggered by a maintainer when the post-launch corpus check regresses or a known-relevant upstream change lands), not on a fixed cadence — automated PR-on-every-upstream-change is deferred until usage data shows it's needed. **The 16/20 recommendation-quality corpus (see Success Criteria) is also a release-blocking gate**: it re-runs on every benchmark-snapshot regeneration; a result below the documented failure threshold (e.g. `< 14/20`) blocks the snapshot from being published as the daily remote copy and auto-files a recalibration issue. Without this enforcement, the "we accept curation responsibility" Key Decision has no measurable teeth after launch.
- **R58.** Recommendations are shown with explicit justification: VRAM fit headroom at the chosen context length, composite benchmark score with source, estimated tok/s with a confidence band, parameter count, quantization variant. Picks must be auditable — no opaque rankings.
- **R59.** The recommender returns the top N (default 3–5) candidates that fit. **The candidate list always includes a first-class "I'll paste a HuggingFace repo id instead" escape option** — not buried behind a flag, not a fallback. Users who already know what they want never need to argue with the recommender. If zero candidates fit at the default context length, the wizard suggests a smaller context, a smaller quant, or skipping the model step rather than recommending something that won't run.
- **R60.** When a user has GGUFs already discovered locally (from R2 cache auto-discovery), the recommender ranks those alongside HuggingFace candidates and labels them `[already on disk]`. Skipping the download step is the natural outcome when a local model already wins.

**Model Download (R46 plumbing)**

- **R61.** All HuggingFace downloads go through the `hf-hub` Rust crate (the official HuggingFace Rust client). No shell-out to `huggingface-cli`; no hand-rolled HF API client.
- **R62.** Downloads land in the OS-specific HuggingFace hub cache directory — Linux: `~/.cache/huggingface/hub`; macOS: `~/Library/Caches/huggingface/hub` — in the canonical `models--<owner>--<repo>/snapshots/<rev>/` form that R2 already scans (see `src/discovery/known_caches.rs::default_huggingface_paths`). R2 auto-discovery surfaces them on the next scan with zero glue code, and other HF-aware tools (transformers, diffusers, `huggingface-cli`) see the same files.
- **R63.** Multi-shard GGUF files are downloaded sequentially using R5's split-GGUF detection over the repo's file listing; resume semantics (range-restart on partial files) are required and must be verified during planning against `hf-hub`'s current behavior, with a thin range-resume wrapper added if the crate doesn't cover it natively.
- **R64.** Disk-space precheck: before any download, the wizard prints required bytes (sum of shard sizes) vs. available bytes on the target filesystem and refuses to proceed when free space < download size + 1 GB headroom. The user can override with an explicit confirmation prompt.
- **R65.** R46's HF-pull primitive — the shared `hf-hub`-based downloader plus its supervised lifecycle — ships as part of init's release because init depends on it. The standalone **`llamastash pull <hf-repo-id-or-url>` CLI subcommand ships in the same release as a functional MVP** (download + progress + checksum verify; not necessarily feature-complete relative to R46's full scope) — required because the tertiary audience (agents driving llamastash via CLI) and the R34 opportunity-cost defense both lean on `pull` being available. The TUI HF-pull hotkey may stage independently in a follow-up release if planning surfaces budget pressure. The primitive's API is the load-bearing commitment; the CLI surface is a committed deliverable; the TUI hotkey is the schedule-flexible surface.

**Config Generation & Tuning**

- **R66.** When config writing is selected, the wizard merges into the user's existing `~/.config/llamastash/config.yaml` (or the path resolved from `LLAMASTASH_CONFIG`) — never overwrites unmodified user-set keys. The user sees a diff preview before any write. Write integrity mirrors v1's `state.json` hardening (see AGENTS.md): the wizard refuses to write if the existing `config.yaml` is a symlink or if `~/.config/llamastash/` is group/world-writable, uses a `config.yaml.tmp.<pid>.<rand>` + atomic rename so concurrent writes can't clobber and a same-UID symlink plant can't redirect, and the final file is `chmod 0600`.
- **R67.** Always-written keys (when not already present): `llama_server_path` (R51 outcome), `port_range` (default `41100..=41300` unless user overrides), `theme` (only if user picks one during the optional R49 step-4 sub-prompt), `probe_timeout_secs` (scaled to detected RAM: 120s default, 240s if RAM < 16 GB). The wizard also persists a `_init_snapshot` block recording detection results at write time — `gpu_vendor`, `vram_gb`, `gpu_device_count`, `llama_server_version`, `init_date`, the verified digest of the installed `llama-server` binary (R53), the snapshot bundle-date used for recommendations, and a `managed_keys` list enumerating exactly which keys the wizard wrote (so re-runs distinguish wizard-set from user-edited keys). This block is the baseline `doctor` (R74) compares against to detect drift and the source of truth re-runs use to decide what they may regenerate.
- **R68.** New schema block `arch_defaults`: per-GGUF-architecture default launch flags keyed by `architecture` field from R7 (`llama`, `qwen2`, `mistral`, `gemma`, `phi`, `qwen3`, etc.). Each block writes the subset of `n_gpu_layers`, `threads`, `cache_type_k`, `cache_type_v`, `flash_attn`, `mlock`, `no_mmap`, `parallel` that the wizard determined sensible for the detected hardware.
- **R69.** Precedence at launch time: per-model preset (R21) > per-model last-launch params (R20) > `arch_defaults` matched by GGUF architecture > built-in defaults. The wizard's writes never override a key the user has already set; the diff preview makes any merge transparent.

**Post-install Verification**

- **R70.** After llama-server is installed or located, the wizard runs a smoke launch in two phases. **Phase 1 — pre-download dry-run:** before committing to a multi-GB download, run R55's VRAM-fit estimation against the chosen model + context and warn (with explicit confirm) if estimated peak is within 10% of the VRAM safety ceiling. **Phase 2 — post-install probe:** spawn `llama-server --version` to confirm the binary executes; then, depending on what step 3 produced and on the active `--only`/`--skip` flags, one of: (a) a freshly downloaded model exists → spawn a real launch against it; (b) no fresh download but R20 has a most-recently-used model on disk → spawn against that; (c) neither → stop at the version check (no spurious pass, no spurious fail). When a real launch happens, init uses the **existing TUI/CLI auto-spawn-on-attach behavior** — IPC `start_model` over the Unix socket; the daemon auto-spawns if missing. No new public non-IPC supervisor entry. Wait for `/health` to return ready, send one tiny chat completion, shut down. Init declares end-to-end success only when whichever phase-2 path runs passes.
- **R71.** Smoke-launch failure modes are surfaced with actionable hints. The VRAM hint covers cases where the safety margin in R55 + R70 phase-1 should have caught it but didn't (rare, e.g. driver/backend overhead unexpected for this GPU class); when R55's filter says "fits" and the model still OOMs at load, that's a genuinely-unpredictable case worth surfacing, not "we recommended this and it doesn't fit." Other actionable hints cover: binary version too old for a required flag, model file appears corrupt (re-pull via `init --only models`), `/health` never went ready within `probe_timeout_secs`, port range exhausted. Network/download failure handling is governed by the originating requirement (R53 for checksum mismatch, R56 for snapshot fetch failure, R63 for partial-shard resume, R78 for `--json` mid-step failure shape). The one new behavior introduced here: **GitHub API rate-limit during R52/R53 retries once with backoff, then falls back to offering the user the "point at an existing binary" path** — this addition isn't covered elsewhere because R52/R53 don't enumerate retry policy.

**Re-run, Idempotency, and `doctor` Sibling**

- **R72.** `init` is fully idempotent. On re-run, each step auto-detects current state and presents skip-or-redo: `✓ llama-server v0.5.2 at /opt/homebrew/bin/llama-server — keep / reinstall / change?`, `✓ Config exists — keep / regenerate?`, `✓ 2 models in catalog — pull another?`. Re-running is the expected path for GPU swaps and llama.cpp upgrades. Config regeneration is **scoped to `_init_snapshot.managed_keys` only**: the wizard regenerates the keys it wrote last time and never touches keys the user has edited or added by hand. `managed_keys` is a list of **dotted paths** (e.g. `['llama_server_path', 'port_range', 'arch_defaults.llama', 'arch_defaults.qwen2']`); leaf-level user edits inside a managed block are preserved via recursive merge, and a managed block no longer relevant (e.g. user removed all qwen2 models) is left in place rather than dropped. `_init_snapshot` itself is special-cased — always rewritten by init, never user-editable; drift on `_init_snapshot.llama_server_path` is the R74-finding-1 signal, not a managed-key concern. The diff preview shows the proposed managed-key changes so the user can audit before confirming.
- **R73.** Targeted re-run: `llamastash init --only server`, `--only config`, `--only models`. Inverse: `llamastash init --skip server`, `--skip config`, `--skip models`. Both flags are repeatable (`--only config --only models`) and accept comma-separated values (`--only config,models`). `--only` and `--skip` are mutually exclusive with each other in the same invocation. **Flag-to-step mapping (pinned):** `server` = R49 step 2 (install or locate `llama-server`); `config` = R49 step 4 (write tuned `config.yaml`); `models` = R49 steps 3 + 5 (download + smoke-launch-against-new-model). R49's step 1 (detection) and step 6 (handoff) always run regardless of flags. R70's phase-2 fallback (see R70) handles the case where `models` is skipped but a smoke probe is still wanted against an existing on-disk model.
- **R74.** `llamastash doctor`: read-only diagnostic. Reuses init's detection code and compares against the `_init_snapshot` baseline in `config.yaml` (R67). **Initial finding set is scoped to 3–5 checks** the team will commit to maintaining; additional checks graduate post-launch only when a real user-reported issue trace would have been shortened by them. The initial set: (1) `llama-server` binary missing or unreadable at the recorded `llama_server_path`; (2) installed `llama-server` binary digest no longer matches the recorded digest **for binaries installed via the GitHub Releases path (R53)** — for brew-installed binaries, doctor trusts brew's integrity model and does not flag version-bump-induced digest changes (avoids training users to dismiss findings on every `brew upgrade`); (3) GPU vendor or VRAM changed since `_init_snapshot.init_date`; (4) benchmark snapshot age (relative to bundle-date stamp, see R56) exceeds 14 days (twice the snapshot's documented soft-staleness budget); (5) config file or its parent directory has unexpected mode (post-R66 hardening drift). Plus one **silent-fallback freshness check** orthogonal to the count cap: if N consecutive `init --only models` runs (or wizard launches) fell back to the bundled snapshot because remote verification failed, doctor surfaces "remote benchmark snapshot has been unreachable since <date>" — this is the detection path for the rollback-DoS attack the Security Contract addendum identifies. Each finding includes a one-line `→ fix with: llamastash init --only X` hint.
- **R75.** `llamastash doctor --json` emits the same findings as a structured object for agent consumption.

**Non-interactive Mode**

- **R76.** `llamastash init --yes` accepts all hardware-aware defaults at every prompt and runs end-to-end without user input. `--yes` implies `--no-confirm` semantics for the config-diff preview.
- **R77.** `llamastash init --json` (combinable with `--yes`) emits a single structured summary at completion: `{installed: {server: {path, version, method}}, downloaded: [{repo, file, bytes, sha}], config: {written_keys, diff}, smoke_launch: {ok, model_id, latency_ms}}`.
- **R78.** Exit codes follow the existing `src/cli/exit_codes.rs` table. Three new codes (pinned):
  - `72` = init aborted, partial state recorded; safe to re-run.
  - `73` = init download failed (network, checksum mismatch, or `hf-hub` error). Distinct from the existing `PULL_FAILED = 69`, which remains the contract for the standalone `llamastash pull` subcommand.
  - `74` = init smoke-launch failed (binary version mismatch, `/health` never ready, OOM at load, port range exhausted).
  These are part of the public CLI contract per `src/cli/exit_codes.rs` and must remain stable across releases.

**Handoff**

- **R79.** Interactive completion ends with a one-paragraph summary block followed by `Launch TUI now? [Y/n]` (default Y). On `Y`, the wizard execs into the TUI with the freshly downloaded (or previously selected) model pre-focused so the user can press Enter on the launch picker. On `n`, prints next-step suggestions (`llamastash list`, `llamastash start <model>`).
- **R80.** Non-interactive completion (`--yes`) prints the summary and exits cleanly. No TUI exec. Agents and scripts continue from there.

## Success Criteria

- A fresh-machine user with no llama-server, no GGUFs, and no llama.cpp knowledge runs `llamastash init`, presses Enter through the recommended defaults, and ends with a healthy OpenAI-compatible endpoint serving a downloaded model. Two paired metrics replace the single end-to-end number: (a) attended-time-to-decision (sum of seconds the user spends answering prompts) < 90 s; (b) wall-clock-to-healthy-endpoint < 15 minutes on a 100 Mbps link for the default recommended model size class. Zero docs read.
- A user who ran `init` 3 months ago and now wants to see if a better-fit model has emerged runs `llamastash init --only models` (or follows the hint from `llamastash doctor`) and gets a refreshed ranking against the latest benchmark snapshot, with existing on-disk models clearly labeled `[already on disk]` so they can decide whether to switch — without re-running the install or config steps.
- An agent runs `llamastash init --yes --json`, parses the structured summary, and knows deterministically whether the install succeeded, what was installed where, and which models were downloaded.
- A user who swaps GPUs (or upgrades llama.cpp from outside llamastash) runs `llamastash doctor`, sees a clear drift report, runs `llamastash init --only server` or `--only config`, and is back in working shape — without losing favorites, presets, or model catalog.
- On a Linux machine with an NVIDIA GPU, `init` defaults to the CUDA prebuilt and never silently installs a CPU-only binary. The user sees the detected GPU + chosen variant in the prompt before any download.
- The recommender's pick for a typical 24 GB VRAM machine surfaces a >= 7B-class quantized model that fits at >= 16k context, with a tok/s estimate that lands within a sensible band of real-world performance. For a CPU-only machine, the pick is a 1–3B Q4 model.
- **Recommendation quality is measured concretely, not informally.** Pre-release, the recommender's top-1 pick must match the maintainer's top-3 manual choice on ≥ 16 of 20 hand-picked `(GPU class, VRAM, task profile)` inputs spanning CPU-only / 8 GB / 16 GB / 24 GB / Apple Silicon. Failures inform either ranker-weight tuning or vendored-data refresh before launch. This corpus is also the regression check against which post-launch Path B-richer-engine investments are justified.

## Scope Boundaries

**Deliberately out of scope for v2 `init`:**
- Installing or updating `llamastash` itself. That remains `cargo install` / `brew install` / GitHub Releases per R45.
- HuggingFace token *creation*. The wizard reads `HF_TOKEN` and `~/.cache/huggingface/token` if present (via `hf-hub`'s standard resolution); it does not orchestrate browser login or token minting.
- Shell `rc` file modification. `init` writes only to `~/.config/llamastash/` and to the resolved `llama_server_path` install location. No `PATH` munging, no shell completions installed automatically (those can come from a separate `llamastash completions <shell>` command).
- llamastash auto-update.
- Distro package managers (apt, dnf, pacman, AUR) as install methods. brew + GitHub Releases + custom-path covers v2; distro packaging is its own coordination problem and tracked separately.
- Windows. Matches v1 R43 — Linux + macOS only.
- Telemetry of any kind. Matches v1 stance.
- Non-GGUF model formats (AWQ, GPTQ, raw HF transformers checkpoints). llamastash consumes GGUFs only.
- A Path B-equivalent richer benchmark layer (multi-source live fetches, confidence grading, lineage recency demotion). The recommender (R55) is deliberately slim — single composite snapshot, single ranker. The richer benchmark layer can land in a follow-up release if usage justifies it.

**Explicit non-features:**
- The wizard does not maintain a llamastash-specific model registry. HuggingFace IS the registry; the recommender ranks against the HF universe; downloads live in the HF cache. Source of truth stays on disk.
- The wizard does not edit user-set config keys. It only writes keys absent from the user's existing config (with a diff preview).

**Opportunity cost (v2 sequencing).** Init is being prioritized ahead of R34 (HTTP / MCP surfaces) in v2. R34 directly serves the agent audience that the project's README explicitly highlights as identity-relevant. Shipping init first means the agent story is held to `init --yes --json` plus the existing R35 CLI surface for the duration of the init release cycle, and meaningful MCP integration slips by that much. The choice is deliberate: closing the first-run "no llama.cpp, no GGUFs" wall is the single biggest blocker to growing adoption for both human users and the agents they'll point at llamastash. R34 lands as the next major v2 increment once init is stable.

## Security Contract (v2 addendum)

v1's security contract (loopback-only Unix socket, `0600` mode, `SO_PEERCRED` auth, no network listener — see AGENTS.md) is preserved unchanged. `init` introduces three new attack-surface categories — binary download + execve, network egress, and HF-token read — each with an explicit control:

**Network egress.** `init` makes outbound calls to three classes of endpoint: (a) the GitHub Releases REST API + asset CDN for llama.cpp prebuilts (R52/R53), (b) the benchmark-snapshot host (R56), (c) HuggingFace Hub via `hf-hub` (R61). Every call obeys the R56 / R53 fetch-contract rules (HTTPS only, allowlisted hosts, capped redirects, refusal of redirects to private/loopback/link-local/metadata IPs, capped body size). `LLAMASTASH_OFFLINE=1` (env var) and `llamastash init --offline` disable every outbound call; `init --offline` downgrades affected steps to "skipped, see hint" and `doctor` plus `init --only config` MUST continue to function fully offline.

**Binary integrity, install path.** R53 governs the GitHub Releases path: HTTPS, SHA-256 minimum, checksum fetched via the GitHub Releases REST API (not HTML scraping), abort-before-extract on mismatch or unavailable checksum, recorded digest in config for `doctor` re-verification, versioned `$XDG_DATA_HOME/llamastash/llama-cpp/<version>/` extraction with `0700` mode, archive-entry safety (refuse `..` / absolute / device / out-of-tree symlinks), atomic extract via `<dir>.tmp.<pid>` + rename.

**Binary integrity, "point at existing" path.** When R51's option 3 or R54's auto-pre-select chooses an existing `llama-server` path, the wizard rejects (with an actionable error) any of: a path under a group/world-writable directory on Linux, a path whose target is itself group/world-writable, a symlink that crosses UIDs. On accept, the wizard computes and persists the SHA-256 digest of the chosen binary in `_init_snapshot` so `doctor` can flag post-acceptance drift.

**HF_TOKEN redaction.** `hf-hub` resolves `HF_TOKEN` from env or `~/.cache/huggingface/token` per its standard contract. Llamastash never writes the token value to stdout, the `init --json` summary, log output, doctor findings, or any error message — only the boolean fact "an HF token is in use" and the masked prefix `hf_***` if mention is unavoidable. The wizard also refuses to use `~/.cache/huggingface/token` when its mode is group- or world-readable; the diagnostic suggests `chmod 600` as the fix. `HF_TOKEN` is not propagated into the spawn env of `llama-server` children.

**Config write integrity.** R66's contract mirrors v1's `state.json` hardening: refuse symlinks at the config path, refuse group/world-writable parent directory, `config.yaml.tmp.<pid>.<rand>` + atomic rename, final mode `0600`.

**Doctor output redaction.** Token values are never included in any `doctor` finding regardless of channel — this is an absolute rule, not a per-finding flag. Given R74's finding set is enumerable (3–5 checks plus the silent-fallback check), agent consumers can hardcode handling against the known finding IDs rather than reasoning about per-record sensitivity. If a future finding genuinely needs differentiated redaction guidance, a `safe_to_log: bool` per-finding tag is introduced *then*, against a real consumer.

**Threat model (top three exploits if implemented without the controls above).** (1) Tampered `llama-server` binary installed via misparsed or missing checksum → persistent RCE every TUI launch. (2) MITM or repo-compromise of the benchmark JSON → users steered to attacker-controlled HF repos and downloading attacker-supplied GGUFs. (3) HF_TOKEN echoed into `init --json` output or doctor findings on a shared CI host → token exfiltration. Each is addressed by a specific control above.

## Key Decisions

- **Ship a curated recommender, accept curation responsibility.** The recommender (R55–R60) is the most consequential identity bet in this release. Mechanical pieces of init (binary install, file download, config write) are neutral plumbing; ranking models is *editorial behavior* — llamastash is telling users which models are good for their hardware. That is registry behavior even though HuggingFace remains the registry. We accept this responsibility deliberately: maintained benchmark snapshot (R56/R57), audit-friendly per-pick justification (R58), local-already-on-disk preferred when it wins (R60), first-class "paste an HF repo id" escape so users who know what they want never have to argue with us (R59). Identity remains "transparent launcher" by virtue of those guarantees — but it shifts from "transparent launcher" to "transparent launcher with an opinionated on-ramp." The on-ramp earns its keep only as long as picks are good (per the recommendation-quality success criterion).
- **Pull R46's primitive forward; let the surrounding surfaces stage.** Init's value proposition requires model downloads, so R46's `hf-hub`-based HF-pull primitive ships in this release. The standalone `llamastash pull` CLI and the TUI HF-pull hotkey ship as MVP slices in the same release if planning has budget; otherwise they stage independently shortly after. Init's ship date is decoupled from R46's full UX (R65). Splitting the primitive would mean re-implementing it twice — that's the part that mustn't split.
- **Terminal-prompts surface, not a TUI page.** Init runs *before* the user has invested in the TUI. Terminal prompts work over SSH, are easy to test deterministically, play well with `--yes` for agents, and avoid forcing ratatui startup before the user has chosen to engage with it.
- **Slim Rust-native recommender (Path A) over shell-out to whichllm.** Single-binary promise (R45), cross-platform parity, no Python runtime dep, no CLI-surface coupling. We pay ~500–700 LOC for a recommender that handles 80% of whichllm's quality for an init wizard's purposes. The benchmark layer is deliberately separated so a richer Path B can layer in later without re-architecture.
- **`hf-hub` Rust crate, not `huggingface-cli` shell-out, for HF integration.** Officially maintained by HuggingFace; cache-layout-compatible with Python `huggingface_hub` (so R2 auto-discovery feeds for free); same auth/env story; async/tokio native. Covers R46 downloads. (The recommender uses the bundled benchmark snapshot R56 as its candidate universe; it does not call hf-hub for search.)
- **CI-refreshed benchmark JSON hosted in the repo, with bundled fallback.** Daily freshness without a service to operate. The aggregation script runs in CI in Python and can lean on whichllm's MIT-licensed `benchmark_sources/` code with attribution; the binary stays single-purpose and pure-Rust.
- **Hardware-aware install default, not brew-everywhere.** Brew bottles for `llama.cpp` on Linux are CPU-only. The wizard never silently installs a CPU-only build on a GPU machine; it picks GitHub Releases with the correct GPU variant when R44 detects a GPU on Linux. Brew is offered only when it's actually the best path or when explicitly preferred.
- **Idempotent + componentized re-run is a first-class requirement, not a polish item.** Init must work as ongoing maintenance for GPU swaps and llama.cpp upgrades. `--only` / `--skip` flags are part of the v2 surface, not deferred.
- **`doctor` ships in the same release as `init`.** Same detection logic, ~20% additional code, large diagnostic and agent-self-healing value. Splitting them costs more than shipping them together.
- **No premature library extraction.** The recommender is a clean internal module inside the llamastash crate. Extraction to a published crate is mechanical the day a second consumer (kdash, another LLM TUI) asks; until then, internal-module velocity wins.
- **MIT-license alignment for vendored whichllm data.** Algorithmic logic is re-implemented in Rust; curated data tables (lineage versions, benchmark source mappings, quantization tables) can be vendored verbatim under MIT with attribution in `NOTICE`. Halves the work without violating the license posture.

## Dependencies / Assumptions

- v1 requirements (hard dependencies of init's own flow): **R2** (HF cache auto-discovery), **R5** (split-GGUF detection), **R7** (GGUF metadata parsing), **R8** (KV-cache-aware VRAM estimation), **R10** (llama-server resolution order), **R16** (health-probed supervisor lifecycle), **R44** (multi-vendor GPU detection), **R45** (single-binary distribution), **R46** (HuggingFace pull — pulled forward into this release).
- v1 requirements consumed by R69's precedence rule when both are present (not hard dependencies of init itself): **R20** (per-model last-launch params), **R21** (per-model presets). Init does not write to R20/R21's data; R69 only observes them during a launch.
- External: `hf-hub` Rust crate at v1.0+ (MIT/Apache-2.0). `dialoguer` or equivalent for terminal prompts. llama.cpp GitHub Releases asset naming remains stable enough to parse (current layout: per-platform per-variant zip archives with checksums published in the release body). Homebrew formula `llama.cpp` continues to ship Metal-enabled bottles on macOS Apple Silicon.
- Vendored: whichllm (MIT) — algorithm reference, data tables, CI script. Attribution recorded in `NOTICE`.
- Schema extension: R67–R69 require extending `src/config/loader.rs::Config` with typed `_init_snapshot: Option<InitSnapshot>` and `arch_defaults: BTreeMap<String, ArchDefaults>` fields. The daemon binary is the canonical reader for `arch_defaults` (R69 precedence applies at launch time, daemon-side); `_init_snapshot` is read by init and doctor only, daemon ignores it. The existing `#[serde(default)]` without `deny_unknown_fields` means wizard-written blocks survive parsing in older binaries but are not observable until the schema lands.
- Assumption: HuggingFace cache layout (`~/.cache/huggingface/hub/<org>--<repo>/snapshots/<sha>/`) stays compatible with the format `hf-hub` writes and that R2 already reads. If HuggingFace introduces a breaking cache-layout change in either client, both R2's discovery and `init`'s downloads must adapt together.
- Assumption: a GitHub Action with default-tier minutes can complete the daily benchmark refresh within job time/memory limits. If sources grow such that this no longer holds, the refresh cadence becomes weekly and the snapshot staleness warning threshold (R74) is relaxed in lockstep.
